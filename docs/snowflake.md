# 雪花算法（Snowflake）原理与设计实现

## 一、背景

在分布式系统中，多个节点需要独立生成全局唯一ID，常见方案有：

| 方案        | 优点           | 缺点                |
|-----------|--------------|-------------------|
| UUID      | 简单，无需协调      | 128位，无序，不适合做数据库主键 |
| 数据库自增     | 有序，简单        | 单点瓶颈，扩展性差         |
| Redis原子自增 | 有序，性能高       | 依赖外部存储            |
| 雪花算法      | 有序，高性能，无外部依赖 | 依赖系统时钟            |

Twitter于2010年开源的雪花算法（Snowflake）以其**有序、高性能、去中心化**的特点被广泛采用。

---

## 二、ID结构

雪花算法生成一个 **64位整数**，最高位为符号位（固定为0），其余 63 位按以下方式划分：

```
 63      62                22  21        17  16       12  11          0
  |        |                |   |         |   |        |   |          |
  0  [ 41位时间戳（毫秒）]   [5位workerID] [5位dataID]  [  12位序列号  ]
```

| 段        | 位数     | 说明                   |
|----------|--------|----------------------|
| 符号位      | 1 bit  | 固定为0，保证ID为正数         |
| 时间戳      | 41 bit | 当前毫秒 - 自定义纪元，可使用约69年 |
| workerID | 5 bit  | 工作节点编号，范围 [0, 31]    |
| dataID   | 5 bit  | 数据中心编号，范围 [0, 31]    |
| 序列号      | 12 bit | 同一毫秒内自增，范围 [0, 4095] |

**理论最大吞吐量**：每个节点每毫秒最多生成4096个ID，即单节点 **409.6万ID/秒**。全集群最多支持 32 × 32 = 1024个节点，整体峰值超过**40亿ID/秒**。

## 三、ID 生成逻辑

```
获取当前毫秒时间戳 ts
          │
          ▼
   ts < lastTs？ ──是──▶ 时钟回拨，返回错误
          │
          │ 否
          ▼
   ts == lastTs？ ──是──▶ seq++
          │                  │
          │                  ▼
          │            seq 溢出（>4095）？──是──▶ 自旋等待下一毫秒
          │ 否               │ 否
          ▼                  │
       seq = 0               │
          │◀─────────────────┘
          ▼
    lastTs = ts
          │
          ▼
  拼装ID并返回
```

### 位拼装

```
ID = (ts - twepoch) << 22
   | dataID          << 12
   | workerID        << 17
   | seq
```

## 四、关键设计细节

### 4.1 自定义纪元（twepoch）

标准Unix毫秒时间戳约为`1.7 × 10¹²`，直接使用会在41位中很快耗尽。通过减去一个自定义起始时间（本项目为`1609000000000`，约2021年初），
将时间戳数值大幅压缩，理论上可从该起点起再使用约**69年**。

### 4.2 同毫秒序列号自增

同一毫秒内连续请求时，序列号从0自增到4095。超出后自旋等待（busy-wait）直到时间推进到下一毫秒，避免ID重复。

### 4.3 时钟回拨处理

依赖系统时钟是雪花算法最主要的风险点。当检测到`now < lastTs`时，本实现**直接返回错误**，由调用方决定如何处理（重试、降级等），不静默生成可能重复的ID。

> 常见的进阶处理策略：回拨幅度小时短暂等待；回拨幅度大时启用备用序列段或告警。

### 4.4 并发安全

使用`sync.Mutex`对`lastTs`和`seq`的读写加锁，保证同一生成器实例在多goroutine下不会产生重复ID。由于临界区极短（无IO、无阻塞），锁竞争开销可忽略不计。

## 五、本项目实现

源码位于 `pkg/utils/id.go`。

```
// 常量定义
const (
    workerBits = 5
    dataBits = 5
    seqBis      = 12
    maxWorker = -1 ^ (-1 << workerBits) // 31
    maxData = -1 ^ (-1 << dataBits) // 31
    maxSeq = -1 ^ (-1 << seqBis) // 4095
    timeShift   = workerBits + dataBits + seqBis // 22
    workerShift = dataBits + seqBis // 17
    dataShift = seqBis              // 12
    twepoch = 1609000000000
)
```

### 使用方式

**自定义节点**：

```
sf := utils.NewSnowflake(workerID, dataID) // workerID, dataID ∈ [0, 31]
id, err := sf.NextID()
if err != nil {
    // 处理时钟回拨错误
}
```

**使用包级默认生成器（workerID=1, dataID=1）**：

```go
id, err := utils.GenID()
```

## 六、局限性与注意事项

1. **时钟依赖**：NTP同步导致的时钟回拨会使生成暂时失败，部署时建议关闭NTP步进同步，改用slew模式。
2. **节点ID管理**：workerID和dataID需在部署时手动分配或通过注册中心（如 ZooKeeper、etcd）动态分配，避免不同节点使用相同组合导致ID冲突。
3. **单调性范围**：ID仅在**同一节点内**严格单调递增；跨节点的ID大小无法保证顺序。
4. **时间戳精度**：精度为毫秒，同毫秒内ID大小由序列号决定，在数据库中按ID排序时近似等于按时间排序。
